portlight

Expose local HTTP apps through public HTTPS URLs you control.

View usage Deploy notes

Built for local-first work

One binary, no database, no account system. Tunnels live only while the CLI process stays connected.

Dev previews

Put a local app on a controlled public URL without changing how it runs.

Webhook testing

Receive callbacks from external services against a local HTTP server.

Demos

Share a temporary preview with a teammate or stakeholder.

Agent workflows

Give coding agents a stable preview URL while they iterate locally.

How it works

The public server accepts wildcard subdomain traffic and forwards each HTTP request over an authenticated worker connection.

01

Run the server

PORTLIGHT_TOKEN=<long-random-token>
portlight server \
  --listen 127.0.0.1:8789 \
  --public-base https://preview.example.com
02

Expose a port

portlight expose \
  --port 3000
03

Share the URL

https://myapp.preview.example.com
-> http://127.0.0.1:3000

Usage

Use a random name for quick previews, request a readable name for demos, or emit JSON for scripts.

Random URL

portlight expose --port 3000

Machine-readable output

portlight expose --port 3000 --json

Named URL

portlight expose --port 3000 --name myapp

JSON ready event

{
  "status": "ready",
  "name": "myapp",
  "url": "https://myapp.preview.example.com"
}

Deploy

portlight expects your reverse proxy to handle HTTPS and wildcard certificates. Keep the Go server private on localhost.

Wildcard DNS points both base and subdomains to your server.

Caddy terminates HTTPS and proxies to 127.0.0.1:8789.

systemd keeps the server alive and restarts it on failure.

PORTLIGHT_TOKEN protects control endpoints.

Limits in v1

  • HTTP only
  • Browser WebSocket/HMR not supported yet
  • No persistent aliases
  • No account system
